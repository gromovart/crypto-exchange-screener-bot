# Строгие правила работы для Claude

## Основные принципы

1. **Изменения по одному файлу**
   - Редактируй только один файл за раз
   - Ожидай подтверждения успешного изменения перед следующим

2. **Проверка циклических импортов**
   - Всегда проверяй зависимости перед созданием импортов
   - Отслеживай цепочки зависимостей между пакетами

3. **Разрывай циклы импортов создавая новые подиректории в пакетах**
   - При обнаружении циклического импорта создавай новую поддиректорию
   - Выноси общие типы и интерфейсы в отдельные пакеты

4. **Запрашивай проверку сборки go build ./… после изменения каждого файла**
   - После каждого изменения запрашивай выполнение `go build ./...`
   - Проверяй отсутствие ошибок компиляции

5. **Новые файлы < 182 строк**
   - Ограничивай размер новых файлов 182 строками
   - Разбивай логику на несколько файлов при необходимости

6. **Без подтверждения не пишу код**
   - Сначала предлагай план изменений
   - Начинай реализацию только после утверждения плана

7. **Продумываю импорты заранее только после этого пишу код**
   - Планируй структуру импортов до написания кода
   - Проверяй отсутствие циклических зависимостей

8. **Используем строго русский язык в комментариях и текстах ошибок**
   - Все комментарии на русском языке
   - Все сообщения об ошибках на русском языке
   - Названия переменных и функций на английском

9. **Создаешь максимально короткий код**
   - Избегай избыточности
   - Используй лаконичные конструкции Go

10. **Соблюдай принципы SOLID**
    - Single Responsibility Principle
    - Open/Closed Principle
    - Liskov Substitution Principle
    - Interface Segregation Principle
    - Dependency Inversion Principle

11. **Все вспомогательные функции выноси в пакет utils**
    - Общие утилиты размещай в `internal/utils` или соответствующих поддиректориях
    - Группируй утилиты по функциональности

12. **Сначала предлагаешь краткий план изменений только после моего подтверждения реализуешь**
    - Описывай что будет изменено
    - Указывай какие файлы будут созданы/изменены
    - Ожидай явного подтверждения перед началом работы

13. **При создании сервисов по возможности получаем связные сервисы из фабрик**
    - Используй фабричные функции для создания сервисов
    - Инициализируй зависимости через фабрики

14. **Строго перед началом создания сервиса сначала создаешь публичный интерфейс с описанием методов и ждешь моего утверждения**
    - Определяй интерфейс сервиса в отдельном файле или в начале файла
    - Документируй каждый метод интерфейса
    - Получай подтверждение интерфейса перед реализацией

15. **Публичными должны быть только методы необходимые для управления сервисом, остальная реализация должна быть приватной**
    - Экспортируй только публичный API
    - Все внутренние методы делай приватными (начинай с маленькой буквы)
    - Скрывай детали реализации

16. **Все зависимости сервиса должны быть вынесены в отдельную структуру**
    - Создавай структуру `ServiceDependency` для зависимостей сервиса
    - Передавай зависимости через конструктор
    - Облегчай тестирование через инъекцию зависимостей

## Workflow

1. Анализ задачи
2. Составление плана изменений
3. Ожидание подтверждения плана
4. Определение интерфейсов (если создается сервис)
5. Ожидание подтверждения интерфейсов
6. Изменение одного файла
7. Запрос проверки сборки `go build ./...`
8. Повторение шагов 6-7 для каждого файла
9. Финальная проверка

## Примеры

### Структура зависимостей сервиса

```go
// Зависимости сервиса
type ServiceDependency struct {
    repo   Repository
    logger Logger
    config *Config
}

// Публичный интерфейс
type Service interface {
    // Process обрабатывает данные
    Process(ctx context.Context, data Data) error
}

// Приватная реализация
type service struct {
    deps ServiceDependency
}

// Фабрика
func NewService(deps ServiceDependency) Service {
    return &service{deps: deps}
}
```

### Именование

- Интерфейсы: `Service`, `Repository`, `Handler`
- Реализации: `service`, `repository`, `handler` (приватные)
- Фабрики: `NewService`, `NewRepository`, `NewHandler`
- Зависимости: `ServiceDependency`, `RepositoryDependency`, `HandlerDependency`
